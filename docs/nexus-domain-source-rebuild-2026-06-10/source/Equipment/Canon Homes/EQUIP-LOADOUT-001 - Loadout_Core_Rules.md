# Loadout Core Rules

## 1. Purpose

This document defines the current working loadout model. It is meant to be usable by DM Mode without becoming a final equipment balance catalog.

## 2. Current baseline loadout

The current baseline loadout is:

```text
Primary Weapon
Secondary Weapon
Tool
Protective Gear
Shield / Defense Module
two Accessory slots
capped consumables
credentials stored separately if acquired
```

This updates and supersedes the older simple model:

```text
Weapon 1
Weapon 2
Armor
Accessory 1
Accessory 2
```

The old model is useful as a history note but should not be used as the active loadout pattern.

## 3. Combat core plus mission overlay

Use a two-layer idea:

### Combat Core

The character's reliable baseline kit. It usually contains the weapon, protective gear, tool, cyberware, and role-defining items they expect to use frequently.

### Mission Overlay

The node-specific preparation layer. It includes tools, accessories, counters, credentials, disguises, breach kits, route aids, social equipment, hacking aids, medical kits, environmental protection, or faction-specific gear selected because the crew knows or suspects what the node needs.

This model makes planning matter without forcing players to track a giant carried inventory.

## 4. Pre-node timing

Loadout selection happens before the away mission / node begins, after enough route or node context exists for planning.

In current DM play, the procedure is:

1. Display current sheets and current loadouts.
2. Apply level-up or advancement if that is happening at the node boundary.
3. Select away team.
4. Select loadouts from shared armory, owned gear, credentials, and scenario access.
5. Confirm node start.

Do not start the node until team, advancement, and loadout are clear unless the scene intentionally denies preparation.

## 5. No carried-but-inactive gear by default

Default rule:

```text
If it is slotted, it is available.
If it is not slotted, it is on the ship/shared armory and unavailable during the node.
```

Exceptions require scenario permission, a specific power, a body/chassis feature, a drone, a cache, a smuggling method, or a written item rule.

## 6. Loadout as modifier and permission source

Loadout can:

- reveal options;
- grant permission to attempt an approach;
- improve checks;
- reduce risk;
- alter partial-success outcomes;
- change AP, MP, reaction, or micro-interaction surfaces;
- create or remove hack surfaces;
- provide tags for combat, social, legal, tech, environmental, or faction play.

DM Mode should treat loadout as a legitimate source of modifiers alongside skill rank, trait, crew assist, prior planning, status, environment, cover, and known advantage/disadvantage.

## 7. Combined-effect powers and tools

A character should not be framed as manually doing two full actions at once.

If a power, tool, cyberware module, or loadout effect combines effects, fiction should frame it as:

- automated;
- preloaded;
- triggered;
- integrated;
- assisted;
- queued;
- linked;
- passive.

This preserves the action economy while allowing gear and powers to feel advanced.

## 8. Mass-intake clarifications

### 9.1 Standard Fit and gear compatibility

Common gear assumes **Standard Fit** unless tagged otherwise. This includes ordinary suits, armor, restraints, crash couches, medkits, doors, tools, emergency systems, and most station/ship interfaces.

Bodies or chassis outside Standard Fit may require adapted gear. Examples include large splicer bodies, nonhuman-scale sleeves, unusually heavy cyberware, unusual integrated limbs, and bodies with special environmental or mobility assumptions.

### 9.2 Integrated gear and slots

Integrated gear is not free extra loadout by default. An integrated tool, weapon, shield, sensor suite, or module must do one of the following:

- occupy a normal loadout slot;
- modify a normal loadout slot;
- count as the character's version of a normal slot;
- explicitly bypass a slot because a cyberware, body, discipline, gear tag, or scenario rule says it does.

This keeps cyberware and body features meaningful without allowing them to silently multiply loadout capacity.

### 9.3 No carried inactive gear

The active rule is:

```text
No carried inactive gear by default.
```

If gear is not in a loadout slot, installed, integrated, carried by a permitted container rule, or granted by a specific scene/scenario rule, it is not available during the node.

### 9.4 Pickups

Pickups are inventory events, not loadout exceptions by default.

A pickup can be:

- added to ship inventory after the node;
- carried as a scene object if the fiction allows;
- used immediately if the scene grants that permission;
- converted into evidence, salvage, crafting material, credential leverage, or route reward;
- equipped later during a valid loadout/preparation window.

A pickup does not automatically create an extra active equipment slot.


## 9. Lattice and durability terminology note

Use full terms in prose when referencing durability surfaces:

```text
Health | System Integrity | Defense | Firewall | Mitigation | Shield
```

Use compact display only where space matters:

```text
HP/SI | DEF/FW | MTG | SHD
```

Armor remains an equipment category under Protective Gear. Mitigation is the numerical damage-reduction stat. This terminology note does not change the active loadout slots or the no-carried-inactive-gear rule.
