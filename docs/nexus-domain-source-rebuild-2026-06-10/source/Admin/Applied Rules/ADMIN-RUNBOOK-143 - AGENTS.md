# Nexus Source Codex Instructions

## Scope

These instructions apply inside the Nexus `00 Source` area.

This area contains Nexus source documents, operating docs, dashboards/state packages, package folders, manifests, reports, and related project records. Treat it as design/source material, not ordinary code.

## Source role

Nexus source docs are the design authority for mechanics, lore, modes, dashboards, package handling, campaign/project state, and prior decisions.

Do not treat file names alone as proof of currentness. Inspect relevant content and metadata before making claims.

## Default behavior

Default to producing reviewable candidate outputs rather than editing live source files.

Direct edits to existing source files require explicit authorization in the task.

Preserve content by default. Do not remove, merge away, overwrite, rename, move, archive, supersede, or delete source material unless explicitly instructed.

When unsure, produce a candidate note/report rather than changing live source.

## Metadata and currentness

Do not call a source, dashboard, package, zip, export, handoff, or replacement current unless the task asked for verification and you checked relevant content and metadata.

When relevant, inspect:
- frontmatter dates and status;
- rev/status fields;
- source role;
- canon status;
- supersedes/superseded-by notes;
- changelog or handling notes;
- README or package manifest.

Do not update dates or rev fields unless content actually changes and the task authorizes that update.

## Package and zip handling

For source package or zip integration tasks:
- list package contents;
- identify README, manifest, changelog, and metadata coverage;
- flag duplicate, stale, missing, conflicting, or unverified files;
- state what the package appears to replace or supplement;
- produce a reviewable integration report or candidate package output.

Do not replace active source packages, delete older packages, or promote candidates without explicit approval.

## Dashboards

Dashboards are mode-owned live transfer/state tools.

Codex may prepare dashboard candidates only when explicitly tasked.

Final dashboard priorities, queues, state language, and publication decisions remain with the relevant Nexus mode.

## Output behavior inside source areas

If an approved review/output path is provided, use it.

If no approved output path is provided, create clearly labeled candidate outputs only in the current working area, or ask for the destination when placement matters.

Do not invent Nexus folder paths.

## Required final report

End each task with:

- source files/packages read;
- candidate files created;
- live files modified, if explicitly authorized;
- metadata/currentness checks performed;
- conflicts, assumptions, or unresolved questions;
- recommended next review step.
