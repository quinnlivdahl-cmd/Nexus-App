# Nexus Source-First Routing Map rev0.1

## 1. Purpose

This map makes source-first operation practical. It tells the assistant which source layer should usually govern common Nexus task types.

This is an internal routing aid, not a required visible response format. The assistant should not turn ordinary answers into source-status boilerplate. It should use this map to avoid re-proposing settled material, ignoring source files, or treating memory as the project database.

## 2. Core Rule

For Nexus project work, active source docs are the default project memory and authority layer. Route through relevant active source docs and domain surfaces before substantive work, including proposals, evaluations, recommendations, drafting, revisions, art direction, Seed watering, DM rulings, playtest execution, source placement, dashboard updates, and implementation planning.

Current user instruction in the active chat remains highest authority. Memory and prior chat summaries are secondary continuity aids. General model knowledge is fallback only.

Surface source uncertainty only when it affects correctness, authority, conflict resolution, currentness, replacement, deletion, or whether a recommendation can safely be treated as source-grounded.

## 3. Routing Ladder

Use this order when forming substantive Nexus output:

1. Current explicit user instruction in the active chat.
2. Bootstrap, Global Mode Rules, and active mode read-first doc.
3. Current routing map and this Source-First Routing Map.
4. Relevant domain README/manifest.
5. Relevant source docs inside the domain.
6. Dashboards/registers when the task concerns current state, work queues, open questions, or active campaign state.
7. Handoffs/passoffs only when uploaded to the active chat, named in a dashboard/index/register, or formally promoted for intake.
8. Memory / recent chat summaries only as continuity context, never as overriding authority.

## 5. Handoff and Passoff Handling

Handoffs/passoffs preserve work and can route material into sources. They are not active authority surfaces by default.

Do not promote handoff files wholesale unless the user asks. Instead:

1. preserve the handoff/passoff in Obsidian handoff folders;
2. extract source-ready decisions into the correct source package or dashboard;
3. leave provisional or rejected material labeled as such;
4. keep deletion protected until verification and explicit approval.

Stewy-to-Stewy handoffs are not default. Use them only for long-chat continuity, real closeout/export, unresolved work that cannot be placed cleanly, or explicit user request.
