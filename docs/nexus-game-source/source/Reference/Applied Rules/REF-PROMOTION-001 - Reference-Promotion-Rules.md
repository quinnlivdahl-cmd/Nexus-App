# Reference Promotion Rules

## Core rule

Reference material is useful evidence, not source truth. A reference item becomes live source only after it is explicitly integrated into the appropriate domain doc, dashboard, data surface, admin runbook, or other accepted canon home.

## Promotion path

1. Identify the reference item and exact file name.
2. Identify the target domain, role bucket, and best destination doc.
3. Check whether active source already covers the point.
4. If the item adds or changes content, route through Steward or Draft depending on depth.
5. Update the target source doc with clear wording, metadata, and handling notes.
6. Record the promotion in the relevant manifest, dashboard/task tracker, or Output Register.
7. Leave the original reference file preserved unless the user approves cleanup.

## Do not promote silently

Do not silently promote:
- old research over current source;
- external inspiration over Nexus design decisions;
- broad lore guidance over active `Lore` canon;
- RNG math over current playtest needs;
- AI runtime research over active operating docs;
- PDF material that has not been reviewed.

## Good promotion examples

- A skill-tree model from a Seed handoff was promoted into the `Skills` domain after user confirmation.
- The cyberware-derived cyborg model was promoted into `Characters` and `Equipment` after user confirmation.
- The Choir/Collective terminology correction was promoted into the `Lore` domain after user correction.

## Reference-use examples

- Use `REF-SURVEY-001` to compare live mechanics against known design patterns.
- Use `REF-REFERENCE-003` to support future Draft work on skills and advancement.
- Use RNG appendices to test probability assumptions before mechanics are finalized.
- Use lore guidance/compendium docs to generate Seed branches, not final canon.
- Use PDF index entries only as pointers until the PDFs are reviewed.
