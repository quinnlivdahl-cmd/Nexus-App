# Nexus RNG Appendix F: Mitigation and Resource Systems
Revision: 1.0  
Status: Draft  
Last Updated: 2026-05-02

## Purpose

This appendix studies systems that let players manage bad luck, push for better outcomes, or convert resources into reliability.

RNG mitigation is important because players should not feel helpless against randomness. It is also dangerous because cheap mitigation can erase risk and flatten the game.

## Canon Control

This appendix does not approve any resource system, reroll system, fail-forward system, or luck-control mechanic.

All mitigation ideas are exploratory until adopted by Core Rules, Combat System, Progression Economy and Resources, or another relevant project document.

## Mitigation Families

| Family | Description | Balance Risk |
|---|---|---|
| Reroll | Roll again after failure or before seeing result. | Can become hidden advantage. |
| Advantage | Roll multiple and keep best. | Strong midrange reliability. |
| Floor | Treat very bad results as a minimum value. | Removes catastrophic failure. |
| Conversion | Spend resource to turn failure into partial success. | Can erase consequence. |
| Push-your-luck | Improve chance or effect while accepting risk. | Risk must be real. |
| Fail-forward | Failure still progresses with cost. | Can make success less meaningful if overused. |
| Banked success | Store future bonuses or successes. | Can create burst turns. |
| Escalating pity | Chance improves after failures. | Good for digital smoothing; may be hard at table. |
| Partial success | Degrees of success create middle outcomes. | Needs clear procedure. |

## Candidate Resource Questions

For any resource that mitigates luck, define:

1. What is the resource called?
2. Who has access to it?
3. How often does it refresh?
4. Can it be generated mid-encounter?
5. Can it be saved across missions?
6. What does it buy?
7. Can it affect other characters?
8. Can it stack with other mitigation?
9. Does it help weak builds or amplify strong builds?
10. What happens when the resource is gone?

## Refresh Models

| Refresh Model | Description | Risk |
|---|---|---|
| Per roll | Always available or very frequent. | Usually too strong. |
| Per turn | Refreshes each round. | Can become routine rather than special. |
| Per encounter | Limited within each fight. | Good for tactical pacing. |
| Per mission | Creates attrition. | Can punish unlucky early rolls. |
| Per route/sector | Strong campaign-level pressure. | May be too harsh for tabletop play. |
| Generated by risk | Earned through danger, heat, stress, or exposure. | Can create degenerate farming. |
| Generated by failure | Comeback/pity resource. | Can feel fair but may reward sandbagging if poorly designed. |

## Reliability vs. Drama

Mitigation should preserve meaningful uncertainty.

A healthy mitigation system may:

- reduce helplessness
- support specialists
- allow clutch decisions
- make failure costly instead of final
- create resource tension

An unhealthy mitigation system may:

- make success nearly automatic
- remove enemy threat
- make specialists better at everything
- make failure bookkeeping rather than drama
- reward hoarding instead of play

## Draft Design Safeguards

Advisory safeguards:

1. Mitigation should usually cost something visible.
2. Rerolls should not stack freely with advantage-like effects.
3. Resource refresh rate matters more than resource name.
4. Mitigation should help more than one archetype, but not all equally.
5. Some enemies or mission conditions may pressure mitigation resources.
6. Partial success can reduce frustration without guaranteeing full success.
7. Luck control should create decisions, not autopilot.

## Update Triggers

Update this appendix when:

- a resource such as Focus, Stress, Heat, Charge, Ammo, Signal Load, Morale, or Battery is proposed
- rerolls, advantage, partial success, or fail-forward rules are proposed
- a build gains repeatable luck control
- playtests show too much frustration from bad rolls
- playtests show too little tension because failure is too avoidable
